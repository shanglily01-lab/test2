## 一、改造背景：为什么需要跨学科改造

超级大脑在工程层面已相当完善：多信号评分、Big4趋势过滤、V2协同验证、黑名单管理，每一层都有充分的逻辑依据。但工程完善≠行为正确。

**问题的本质是：系统在设计上忽略了两类已被生物进化和心理学反复验证的失败模式。**

### 生物学失败模式
生命系统在亿万年进化中发展出了精密的自保机制，而这套交易系统恰好缺失了其中四项关键保护：
- 没有**不应期**（神经元在激活后的保护性静默）→ 可以在同一币种上连续重复犯同一个错误
- 没有**习惯化**（重复无效刺激后反应减弱）→ 对持续亏损的币种毫无戒备
- 系统B的**正反馈失控**（亏损触发马上平仓）→ 短暂波动会误伤正常持仓
- 没有**操纵子**（环境变化触发模式切换）→ 熊市中系统仍然无区别地尝试做多

### 心理学失败模式
人类在高压决策中反复出现六种认知偏误，系统设计时无意中将这些偏误固化进了规则：
- **确认偏误**：信号清洗时直接丢弃反向信号，从不统计有多少证据在反对这笔交易
- **沉没成本**：分阶段超时平仓的逻辑是"因为持了这么久"，而非"因为市场已经告诉我方向错了"
- **群体极化**：资金费率极端时系统仍可开仓，错过了散户情绪过热的最强预警信号
- **从众效应**：Big4强度越高，系统开仓越踊跃，恰好在市场最一致的时刻追顶/追底
- **基本归因错误**：权重优化器被禁用，系统只会把亏损归咎于"市场问题"，从不检讨自身信号权重是否失效
- **羊群效应**：一次扫描出现大量同向信号时，系统照单全收，不质疑这是否是市场情绪化的集体幻觉

## 二、生物学改造（4 项）

### B1 — 动作电位不应期
**生物原理**

神经元在产生一次动作电位后，会进入 1-2 毫秒的绝对不应期——在此期间无论多强的刺激都无法再次激发。这不是设计缺陷，而是保护机制：防止神经元在单次刺激下持续激活，保证信号以离散脉冲形式传导而非持续叫嚣。

**系统问题**

原系统在同一币种同一方向平仓后，可以立刻再次开仓。这意味着如果市场处于震荡格局，系统会在同一价格区间反复开仓、反复止损，形成"手续费收割机"。

**改造实现**

新增 `check_recent_open()` 方法，查询 `futures_positions` 表：

```python
def check_recent_open(self, symbol: str, side: str, cooldown_minutes: int = 120):
    """同 symbol 同方向开仓后 2 小时内拒绝重复开仓"""
    cursor.execute("""
        SELECT COUNT(*) FROM futures_positions
        WHERE symbol=%s AND position_side=%s AND account_id=%s
          AND status='open'
          AND created_at >= DATE_SUB(NOW(), INTERVAL %s MINUTE)
    """, (symbol, side, self.account_id, cooldown_minutes))
```

在 `open_position()` 中，所有其他验证之后加入检查：

```python
if opp.get('signal_type') != 'EMERGENCY_BOUNCE':
    if self.check_recent_open(symbol, side, cooldown_minutes=120):
        logger.warning(f"[REFRACTORY] {symbol} {side} - 开仓不应期内(2小时)，拒绝重复开仓")
        return False
```

**例外设计**：`EMERGENCY_BOUNCE`（Big4触底急跌反弹）信号不受此限制，因为触底反弹本身就是强信号，需要立即响应。

**效果**：同币种同方向连续入场的最短间隔从"无限制"变为 2 小时。

### B2 — 突触习惯化
**生物原理**

突触在受到重复刺激后，释放的神经递质逐渐减少，突触后膜的响应幅度也逐渐降低。这是神经系统最基础的学习机制——对持续无效的刺激"脱敏"，将注意力资源转移到新的、有意义的刺激上。

**系统问题**

原系统对每次信号平等对待。一个在某币种上已经连续亏损 5 次的信号，和从未亏损的信号会受到相同的对待，都只需要达到 60 分即可开仓。系统没有"记吃亏"的能力。

**改造实现**

新增 `get_symbol_loss_multiplier()` 方法，查询最近 5 笔已平仓订单，统计从最新一笔起的连续亏损次数：

```
连续亏损次数 → 系数  → 有效开仓阈值（60 / 系数）
0 ~ 1 次    → 1.0   → 60 分（无惩罚）
2 次        → 0.9   → 67 分
3 次        → 0.8   → 75 分
4 次        → 0.7   → 86 分
5+ 次       → 0.6   → 100 分
```

**重置机制**：一旦出现盈利单，系数立即重置为 1.0（习惯化可逆）。

**效果**：对"惯亏"的币种自动提高准入门槛，迫使系统只在该币种出现真正强力信号时才重新入场。

---

### B3 — 负反馈抑制（System B 稳定化）
**生物原理**

生物系统通过负反馈防止正反馈失控。肾上腺皮质在应激时分泌皮质醇，但皮质醇本身会抑制下丘脑，防止应激响应无限放大。如果没有这个抑制机制，一次轻微压力就会导致激素风暴。

**系统问题**

System B（分阶段超时）的原始逻辑是：持仓超过 X 小时且亏损超过 Y% → 立即平仓。但加密市场的价格波动频繁，一笔持仓完全可能在某个瞬间触及阈值，随后恢复。原来的逻辑会在这个瞬间误触发，砍掉本来会盈利的仓位。

**改造实现**

在 `SmartExitOptimizer` 中新增 `_loss_onset_times` 字典，记录每个档位首次触发亏损的时间戳：

```python
self._loss_onset_times: Dict[str, datetime] = {}  # "position_id:hour_checkpoint" -> 首次触发时间
```

触发逻辑改为两阶段确认：

```
首次触达亏损阈值 → 开始30分钟计时，本轮不平仓
亏损持续 ≥ 30 分钟 → 确认非短暂波动 → 执行平仓
亏损中途恢复 → 清除计时器（负反馈重置）
```

**效果**：分阶段超时从"瞬间触发"改为"持续确认后触发"，假阳性（误平仓）大幅减少。

---

### B4 — 操纵子基因表达调控
**生物原理**

大肠杆菌乳糖操纵子是最经典的基因调控案例：在没有乳糖的环境中，阻遏蛋白关闭乳糖代谢基因，不浪费能量；乳糖出现后，阻遏蛋白被解除，基因立即表达。这是"按需开启"的适应性调控。

**系统问题**

原系统的做多/做空开关（`allow_long` / `allow_short`）仅能手动调整。系统在熊市中会持续尝试做多（对应乳糖不存在时强行消耗乳糖代谢酶），在牛市中会持续尝试做空，不能根据市场环境自动切换策略模式。

**改造实现**

新建 `app/services/big4_regime_monitor.py`，实现三态市场状态机：

```
检测窗口：过去 48 小时的 big4_trend_history 信号分布
判断逻辑：

  BULLISH 信号占比 ≥ 70%  →  BULL 状态   →  allow_long=1, allow_short=0
  BEARISH 信号占比 ≥ 70%  →  BEAR 状态   →  allow_long=0, allow_short=1
  其余情况                →  SIDEWAYS 状态  →  allow_long=1, allow_short=1
```

调用方式：在 `smart_trader_service.run()` 主循环中每小时调用一次 `regime_monitor.run_detection()`，仅在状态变化时写入数据库，避免无效写入。

**效果**：系统具备了自动感知市场大趋势并切换作战模式的能力，彻底消除了"顶风做多"和"逆势做空"的系统性误操作。

## 三、心理学改造（6 项）

### P1a — 确认偏误防御
**认知偏误定义**

人在做决策时，会下意识地寻找和放大支持自己已有判断的信息，同时忽略或否定反面证据。分析师看好某股，就会对利好消息高度敏感，对利空消息视而不见。

**系统问题**

原系统在 `scan_one()` 里，`signal_components` 字典在信号清洗时直接把反向信号丢弃。系统从不问：有多少个信号在反对这笔交易？即使市场有 3 个明确的空头信号正在喊"不要做多"，系统只要多头信号分数够，就会无视这些反对声音。

**改造实现**

在信号清洗**之前**，统计反向信号的数量（因为清洗后这些信号就消失了）：

```python
# 在 scan_one() 信号清洗代码之前
if side == 'LONG':
    counter_signal_count = sum(1 for sig in signal_components if sig in bearish_signals)
else:
    counter_signal_count = sum(1 for sig in signal_components if sig in bullish_signals)
```

将 `counter_signals` 写入返回字典，在 `open_position()` 中进行最终裁决：

```python
counter_signals = opp.get('counter_signals', 0)
if counter_signals >= 3:
    logger.warning(f"[CONFIRMATION_BIAS] {symbol} {side} - {counter_signals}个反向信号存在，拒绝开仓")
    return False
```

**设计意图**：阈值选 3，而非 1 或 2。允许市场存在少量分歧（1-2个反向信号是正常的），但当 3 个或更多独立的技术信号都在说"反向"时，这笔交易的方向判断本身就值得怀疑。

### P1b — 沉没成本防御
**认知偏误定义**

人倾向于因为已经投入的成本（时间、金钱、精力）而继续一个决策，而不是根据未来的合理预期来判断。赌徒不肯离桌，因为"我已经输了这么多，再赌一把说不定能回来"——这是错误的。已经亏掉的钱不该影响下一个决策。

**系统问题**

原 System B 的分阶段超时逻辑本质上是：**因为已经持仓 X 小时了**，才在 Y% 亏损时平仓。这是时间维度的沉没成本——系统的平仓动机是"持的时间够久了"，而非"市场已经告诉我这个方向错了"。

**改造实现**

在 `SmartExitOptimizer` 中新增优先级 5.5 检查（在分阶段超时之前执行）：

```python
# 查询 V2 K线评分的最新方向
SELECT direction, strength_level FROM coin_kline_scores WHERE symbol = %s

# 触发条件：
# 1. 当前亏损（pnl_pct < 0）
# 2. V2 方向与持仓方向完全相反
# 3. V2 强度为 'strong'（排除偶发波动）
if _decay_pnl_pct < 0 and v2_direction == opposite_direction and v2_strength == 'strong':
    return (f'信号衰减(V2强力反转{v2_direction})', 1.0)
```

**设计意图**：不问"持了多久"，只问"市场现在还支持这个方向吗"。当 V2 K线评分已经强力判断方向反转时，无论持仓多久，立即平仓。`strong` 强度要求是为了过滤噪声——只有多个时间周期同时反转才触发。

---

### P2a — 群体极化防御：资金费率否决
**认知偏误定义**

群体在讨论后，往往会得出比个体更极端的判断。金融市场中，当散户情绪高度一致时，往往意味着市场已经被充分定价，机构正在准备反向操作。

**资金费率的信号意义**

永续合约资金费率由多空双方的仓位不平衡决定：
- 资金费率 > 0：多头向空头付费，说明市场上做多的人更多，多头溢价
- 资金费率 > 0.1%：散户情绪极度看多，机构往往在此时大量开空

**改造实现**

新增 `get_funding_rate_pct()` 方法查询 `funding_rate_stats` 表，在 `open_position()` 中加入否决逻辑：

```python
funding_rate_pct = self.get_funding_rate_pct(symbol)
if funding_rate_pct is not None:
    if side == 'LONG' and funding_rate_pct > 0.1:
        logger.warning(f"[FUNDING_VETO] {symbol} - 资金费率{funding_rate_pct:.3f}%>0.1%，散户过度看多，拒绝追多")
        return False
    elif side == 'SHORT' and funding_rate_pct < -0.1:
        logger.warning(f"[FUNDING_VETO] {symbol} - 资金费率{funding_rate_pct:.3f}%<-0.1%，散户过度看空，拒绝追空")
        return False
```

**设计意图**：否决方向是单向的——费率极端看多时禁止做多（不禁止做空），费率极端看空时禁止做空（不禁止做多）。这不是反转策略，而是"不在散户情绪最高点追涨/杀跌"的保守原则。

---

### P2b — 从众效应防御：Big4 众包惩罚
**认知偏误定义**

当绝大多数人都倾向于同一个判断时，个体会倾向于顺从，哪怕内心有疑虑。在市场中，"所有人都看多"通常不是安全入场的信号，恰恰相反——它是接近顶部的信号。

**系统问题**

原系统的 Big4 逻辑是：Big4 强势看多（强度≥50）→ LONG 阈值从 100 降到 60，大幅降低入场门槛。这个逻辑在方向上没有问题（顺势交易），但没有考虑"顺势"和"追顶"之间的区别——大趋势越强，追涨的风险越高，越接近反转节点。

**改造实现**

在 `scan_one()` 的阈值计算中引入分级众包惩罚：

```python
_crowding_penalty = 0
if _b4_strength >= 70:
    _crowding_penalty = 15   # 全市场共振：同向开仓阈值 +15
elif _b4_strength >= 55:
    _crowding_penalty = 8    # 中度群体效应：同向开仓阈值 +8

if big4_bullish:
    long_threshold = 60 + _crowding_penalty   # 牛市: 60 / 68 / 75
short_threshold_adj = self.threshold + (_crowding_penalty if big4_bearish else 0)
```

**各状态对应阈值**：

| Big4 强度 | 市场状态 | LONG 阈值 | SHORT 阈值 |
|---------|--------|---------|---------|
| 普通行情（无Bull/Bear） | 中性 | 100 | 60 |
| BULLISH，强度 50~54 | 弱势牛 | 60 | 60 |
| BULLISH，强度 55~69 | 中度牛（中度从众） | 68 | 60 |
| BULLISH，强度 ≥70 | 强势牛（高度从众） | 75 | 60 |
| BEARISH，强度 55~69 | 中度熊（中度从众） | 100 | 68 |
| BEARISH，强度 ≥70 | 强势熊（高度从众） | 100 | 75 |

**设计意图**：顺势而为，但在大趋势越强时要求信号越强。如果一笔交易在 Big4 强度 72 的情况下只有 68 分，说明它不够强——不值得在市场最一致的时刻追进去。

### P3a — 基本归因错误修正：重启权重自适应
**认知偏误定义**

人倾向于将他人行为归因于内在性格（"他就是这样的人"），而将自己的失败归因于外部环境（"是市场的问题"）。在量化交易中，这表现为：系统亏损时，首先想到的是"市场行情不好"，而不是"我的信号权重可能已经失效"。

**系统问题**

信号权重自适应优化器（`ScoringWeightOptimizer`）此前被手动禁用，注释写着"权重由人工维护"。这个决策有其合理性（防止自动调整破坏人工调优的结果），但副作用是系统彻底失去了自我学习能力——当某个信号组件的真实表现开始下滑时，系统无从知晓，只能继续使用失效的权重。

**改造实现**

在 `adaptive_optimizer.py` 中重新启用调用，并加入完整的异常保护：

```python
# 原来（已禁用）：
if apply_weights:
    logger.info("信号评分权重自动调整已禁用，跳过")

# 现在（重新启用）：
if apply_weights:
    try:
        weight_results = self.weight_optimizer.adjust_weights(dry_run=False)
        # 日志记录调整了哪些组件
    except Exception as _we:
        logger.error(f"权重调整失败（不影响交易）: {_we}")
```

**内置约束**（来自 `ScoringWeightOptimizer` 内部逻辑）：
- 每次最多调整 ±3 分
- 权重绝对范围：[5, 30]，不会出现极端值
- 需要 ≥5 笔样本才会调整对应组件，单笔异常不会影响权重

**设计意图**：不是让机器完全接管权重，而是在人工设定的安全边界内，让 7 天的实际数据对权重进行微调。如果某个信号组件的胜率明显下滑，系统会自动降低它的权重，而不是等待人工干预。

### P3b — 羊群效应防御：从众探测器
**认知偏误定义**

个体在面对不确定性时，倾向于模仿群体的行为——即使群体的判断是错误的。股市崩盘前往往有一段"一致性最高"的时期，所有人都在买入，理由都是"别人都在买"。

**系统问题**

`scan_all()` 会将所有合格信号直接返回给开仓逻辑。如果某一轮扫描出现了大量同向信号（例如 10 个信号中 9 个都是 LONG），系统会照单全收，把这 9 个信号都排队准备开仓。但这种高度一致恰恰可能是市场情绪化追涨的集体幻觉。

**改造实现**

在 `scan_all()` 返回前，对结果集进行群体偏斜检测：

```python
if len(opportunities) >= 5:
    long_count = sum(1 for o in opportunities if o['side'] == 'LONG')
    short_count = len(opportunities) - long_count
    dominant_pct = max(long_count, short_count) / len(opportunities)

    if dominant_pct >= 0.8:   # 80% 以上同向
        dominant_side = 'LONG' if long_count > short_count else 'SHORT'
        herding_threshold = self.threshold + 10  # 60 → 70
        # 过滤掉同向中低于新阈值的信号
        opportunities = [
            o for o in opportunities
            if o['side'] != dominant_side or o['score'] >= herding_threshold
        ]
        logger.warning(f"[HERDING] {dominant_pct*100:.0f}%信号偏{dominant_side}，阈值提高至{herding_threshold}分")
```

**触发条件**：信号数 ≥5 个（避免小样本误判）+ ≥80% 同向（允许正常的趋势性偏斜，只过滤极端情况）。

**效果**：在市场情绪极度一致时，系统不跟随群体，要求同向信号必须更强（≥70分）才能通过。

## 四、整体改造架构

```
开仓决策流程（改造后）
```

## 五、量化影响预估

| 改造项 | 预期影响方向 | 预期效果 |
|--------|------------|---------|
| B1 不应期 | 减少开仓频率 | 减少同一币种的连续错误，降低手续费消耗 |
| B2 习惯化 | 选择性减少开仓 | 对连续亏损的币种提高门槛，减少"越亏越追"行为 |
| B3 系统B稳定化 | 减少误平仓 | 30分钟确认窗口过滤短暂波动触发的假信号 |
| B4 市场状态机 | 减少逆势交易 | 熊市不做多，牛市不做空，方向准确率提升 |
| P1a 确认偏误 | 减少开仓频率 | 过滤市场有明显分歧时的低质量信号 |
| P1b 沉没成本 | 加速止损 | 方向反转且亏损时，不再等超时才平仓 |
| P2a 资金费率 | 减少追高/追低 | 在散户情绪最极端时退出同向交易 |
| P2b 众包惩罚 | 提高强趋势中的门槛 | 在市场最一致时要求更强的信号确认 |
| P3a 权重自适应 | 长期信号质量提升 | 自动降低失效信号的权重，提升整体准确率 |
| P3b 从众探测 | 减少批量追涨/杀跌 | 大量同向信号时过滤低分信号，避免集体失误 |

## 六、日志监控说明

改造引入了 7 个可识别的日志标签，方便运营时快速定位拦截原因：

```
[REFRACTORY]         → 不应期拦截（同向开仓2小时内重复）
[HABITUATION]        → 习惯化拦截（连败阈值不足）
[CONFIRMATION_BIAS]  → 确认偏误拦截（反向信号≥3个）
[FUNDING_VETO]       → 资金费率否决（散户情绪极端）
[CROWDING]           → 众包惩罚提示（Big4强度过高，阈值提升）
[HERDING]            → 从众探测过滤（批量同向信号）
[SIGNAL_DECAY]       → 信号衰减平仓（V2强力反转+亏损）
```

**快速排查示例**：
```bash
# 查看今天有多少笔被各机制拦截
grep -E "\[REFRACTORY\]|\[HABITUATION\]|\[CONFIRMATION_BIAS\]|\[FUNDING_VETO\]|\[HERDING\]|\[SIGNAL_DECAY\]" logs/smart_trader.log | grep "$(date +%Y-%m-%d)"
```

## 七、重启说明

所有改造需重启 `smart_trader_service.py` 后生效。

**重启前请注意**：
1. `_loss_onset_times` 字典（B3 计时器）会在重启时清空，正在计时的分阶段超时会重新计算，不影响最终结果
2. Big4RegimeMonitor 会在第一次主循环（约1分钟后）完成首次检测并写入 `allow_long/allow_short`
3. 现有持仓不受影响，SmartExitOptimizer 会通过 `start_monitoring()` 重新拾取所有 `open` 状态的持仓
